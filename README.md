# AirMic

AirMic 是一套基于 ESP32 的蓝牙语音输入方案，目标是把电池供电的便携设备做成一个可被 Windows 识别和使用的 HFP 麦克风，并在 PC 端完成编码监听、快捷键触发、设备状态检测和音频参数调试。

当前仓库按“设备固件 + Windows 上位机”双端协同路线组织，重点解决这几类问题：

- ESP32 侧把 INMP441 麦克风音频送入 Windows HFP 话路
- 在音频流里嵌入控制音，替代不稳定的额外控制链路
- PC 侧监听 HFP 输入流，解码控制音后触发快捷键
- 在不打开调试串口的情况下，持续判断监听服务、设备在线状态、系统输入状态
- 在需要调试时再通过串口读取和下发设备参数，避免影响正常使用和烧录

## 仓库结构

- `firmware-esp32-hfp`
  当前主线 ESP32 固件。负责 HFP 麦克风、按键输入、控制音注入、音频参数配置与保存。
- `desktop-app`
  当前主线 Windows 桌面应用。基于 PySide6，负责后台监听、快捷键映射、系统音频状态检测、调试面板与设备参数配置。
- `shared`
  共享文档、协议说明和后续可复用资料。

## 技术路线

项目当前采用的是“**Classic Bluetooth HFP 音频链路 + 音频内嵌控制音 + PC 端解码触发**”路线，而不是 BLE 按键链路或额外常驻驱动路线。

选择这条路线的原因：

1. Windows 对 ESP32 这类设备最稳定、最自然的“麦克风接入”方式仍然是 HFP Hands-Free 音频输入。
2. 单独走 BLE/GATT/PTT 控制虽然实现简单，但实际重连、状态保持和 Windows 兼容性不够稳定。
3. 把控制信息直接编码到 HFP 麦克风音频流里，可以让“录音启动 / 扩展按键触发”与音频链路天然同步，减少多链路状态漂移。
4. PC 侧使用普通用户态程序即可完成监听、解码和快捷键发送，不依赖内核驱动。

## 当前实现路径

### 1. 设备端

ESP32 固件负责：

- 采集 INMP441 I2S 麦克风音频
- 做基础音频处理，包括：
  - 采样位移
  - 增益
  - 简单噪声门限
  - 16 kHz 到 8 kHz 下采样
- 通过 Classic Bluetooth HFP HF 角色把音频送入 Windows
- 在音频流里注入控制音：
  - `START`
  - `Tone A`
  - `Tone B`
  - `Tone C`
- 通过串口命令支持运行时调参和保存配置

当前主线按键定义：

- `GPIO13`：Start / PTT
- `GPIO25`：Tone A
- `GPIO26`：Tone B
- `GPIO27`：Tone C

### 2. 上位机

Windows 桌面应用负责：

- 后台启动音频探针，监听 `ESP32-AirMic-HFP` HFP 输入设备
- 从 WASAPI 捕获到的音频流中识别控制音
- 将控制音映射为快捷键动作
- 提供快捷键录制、测试和持久化
- 检测系统默认输入、AirMic 设备状态、后台监听状态
- 在调试面板打开时，通过串口读取 / 下发设备音频参数

### 3. 控制链路

当前主线路由如下：

1. 用户按下 ESP32 按键
2. ESP32 注入对应控制音到 HFP 麦克风流
3. Windows 将 HFP 音频作为普通录音设备暴露
4. 桌面应用后台探针捕获该输入流
5. 探针解码出 `START / A / B / C`
6. 应用把解码结果映射到快捷键动作
7. 最终触发千问 App、微信语音输入或用户自定义快捷键

## 技术栈

### 设备端

- ESP32
- ESP-IDF 5.4.x
- Classic Bluetooth HFP HF
- I2S 麦克风输入
- NVS 持久化配置

### 上位机

- Python 3
- PySide6
- Windows WASAPI
- C# / NAudio 音频探针
- 串口命令通道

## 上位机监听、捕获与状态检测方法

### 1. 音频监听与捕获

上位机不直接依赖 Python 音频回调做核心识别，而是使用独立的 C# 探针程序：

- `AirMicAudioProbe_v5.exe`

其职责是：

- 枚举 Windows 捕获设备
- 等待名称匹配 `ESP32-AirMic-HFP` 的 HFP 输入端点变为 `Active`
- 使用 WASAPI 捕获音频
- 输出 RMS / Peak 等基础音量信息
- 对控制音进行 Goertzel 检测，识别：
  - `START`
  - `Tone A`
  - `Tone B`
  - `Tone C`
- 在无音频回调时触发 watchdog 重启提示

桌面应用通过后台线程启动该探针并读取其标准输出，把探针事件转换成应用内事件。

### 2. 控制音解码

当前控制音使用双频组合编码，PC 侧根据固定低频和高频组合识别事件。

当前已使用的事件为：

- `START`
- `A`
- `B`
- `C`

解码结果最终进入桌面应用的动作映射层，触发主语音键或扩展快捷键。

### 3. 设备状态检测

设备状态检测不是只看串口，也不是只看音频设备，而是综合多路信息：

- `AirMicAudioProbe_status.exe` 持续监听 Windows 音频端点状态
- 读取默认通讯输入与多媒体输入
- 读取 AirMic HFP 端点是否存在、是否为 `Active`
- 监听 PnP / 蓝牙相关变化提示
- 结合蓝牙连接迹象判断设备是否真正在线

桌面应用内部会把这些信号整理为三个面向用户的状态：

- 监听服务状态
- 设备状态
- 系统输入状态

### 4. 串口状态检测

串口只在调试页打开时使用，主要用途是：

- 读取当前设备音频参数
- 下发临时参数
- 保存参数到设备 NVS
- 发送测试命令

串口**不作为主界面设备在线状态的唯一依据**，避免误判，也避免后台长期占用串口影响烧录。

## 当前主要功能

### 固件侧

- HFP 麦克风音频输入
- Start / Tone A / Tone B / Tone C 控制音注入
- 录音门控模式切换
- 麦克风增益、噪声门、控制音增益、位移参数调节
- 配置保存到 NVS

### 桌面应用侧

- 后台自动启动监听
- 快捷键录制与测试
- 快捷键设置持久化
- 监听服务状态显示
- 设备状态显示
- 系统输入状态显示
- 调试台串口控制

## 已知边界

- 当前仍基于 HFP 窄带音频，音质与专业 USB 麦克风方案不同。
- Windows 对 HFP 自动重连是否接受，仍受本机蓝牙适配器、驱动和系统策略影响。
- 如果设备完全断电再唤醒，蓝牙重连和音频链路建立仍然需要时间。
- 当前方案优先解决“可用、稳定、易恢复”，而不是极限低功耗或极限低时延。

## 建议使用方式

1. 先烧录 `firmware-esp32-hfp`
2. 与 Windows 完成蓝牙配对
3. 启动 `desktop-app`
4. 等待 HFP 输入设备被探针捕获
5. 在主界面配置快捷键映射
6. 需要调参时再打开调试台和串口
